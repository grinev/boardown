
# Архитектурные рекомендации (Architectural Recommendations)

Основываясь на анализе текущей кодовой базы (монорепозитория с пакетами `core`, `ui` и `web`), проект имеет достаточно хорошую и чистую архитектуру для локального таск-менеджера.

## 🏗️ Текущая архитектура и её сильные стороны

1.  **Четкие границы (Monorepo):** Пакеты хорошо разделены. `core` содержит доменную логику, парсинг и схемы; `ui` отвечает за отображение и состояние (Zustand + React); `web` связывает всё вместе (Vite, адаптеры).
2.  **Инверсия зависимостей (Dependency Injection):** Использование `FsAdapter` — отличное решение. Оно отвязывает ядро от конкретной реализации файловой системы, что позволяет легко подменять реализацию для веба (`dev-http-fs-adapter.ts`) или тестов.
3.  **Безопасность типов:** Использование `Zod` (в `schemas.ts`) для валидации данных, приходящих из файлов, гарантирует надежность доменных моделей на раннем этапе.
4.  **Чистые функции:** В `board-ops.ts` логика изменения состояния (создание, перемещение задач) отделена от побочных эффектов записи.

---

## 🚀 Архитектурные улучшения (пока проект не разросся)

Чтобы избежать технического долга по мере роста функционала, рекомендую внедрить следующие изменения:

### 1. Переход на AST-парсер для Markdown (`packages/core/src/parser.ts`)
*   **Проблема:** В данный момент парсинг файлов (разделение frontmatter и тела, парсинг задач) опирается на ручную обработку строк (`splitBody`, регулярные выражения).
*   **Решение:** По мере того как пользователи начнут использовать более сложное форматирование в Markdown, ручной парсинг станет хрупким. Рекомендуется перейти на AST-ориентированный подход (например, используя экосистему **unified / remark / mdast**). Это сделает чтение и, что более важно, безопасную модификацию файлов (сериализацию) гораздо надежнее.

### 2. Разделение UI-состояния и логики сохранения (`packages/ui/src/store.ts`)
*   **Проблема:** Store сейчас выступает в роли оркестратора: он управляет и локальным состоянием UI (например, тема оформления), и отвечает за загрузку/сохранение доменных данных (`load`, вызовы адаптера).
*   **Решение:** Вынести логику оркестрации (persistence) в отдельный слой (например, `BoardService` или контроллер в пакете `core`). Store должен оставаться "глупым" хранилищем данных для React. Это предотвратит разрастание файла хранилища и упростит обработку гонок данных (race conditions) при одновременном сохранении нескольких файлов.

### 3. Атомарность операций (Транзакции)
*   **Проблема:** Функции вроде `moveTaskBetweenContainers` в `board-ops.ts` потенциально требуют обновления нескольких файлов (например, удаление из одного релиза и добавление в другой).
*   **Решение:** На уровне инфраструктуры (вокруг `FsAdapter`) стоит продумать механизм "пакетного сохранения" или простых транзакций. Если запись первого файла прошла успешно, а второго упала (например, из-за прав доступа в `dev-fs-plugin.ts`), система должна уметь корректно обрабатывать такие частичные отказы.

### 4. Конфигурация путей (`packages/core/src/loader.ts`)
*   **Проблема:** Наличие жестко закодированных путей (`RELEASES_DIR`, `EPICS_DIR`) внутри загрузчика. Также здесь выполняется сайд-эффект по обновлению `nextId` в `config.yaml`.
*   **Решение:** Пути к директориям должны приходить извне (например, из конфигурационного файла, который считывается на этапе инициализации приложения). Управление ID лучше полностью инкапсулировать в `id-generator.ts` и его собственный сервис, чтобы `loader` занимался только загрузкой домена.

### 5. Индексация данных для UI (`packages/ui/src/utils/`)
*   **Проблема:** Поиск задач (например, `find-tasks-by-epic.ts`) при росте количества эпиков и релизов будет работать за `O(N)`.
*   **Решение:** При загрузке данных в store стоит формировать нормализованные структуры данных (например, словари `Record<TaskId, Task>` и `Record<EpicId, TaskId[]>`), чтобы поиск и получение связанных данных происходили за `O(1)`.
